# Spring Boot + Vue3 用户认证系统 — 构建教程

## 项目概述

本项目是一个 **前后端分离** 的用户注册、登录、鉴权系统：

- **后端**：Spring Boot 3.5.11 + MyBatis-Plus + MySQL + Redis + JavaMail（163邮箱）
- **前端**：Vue 3 + Vite + Element Plus + Vue Router + Axios

核心特色是 **图形验证码防刷 + 邮箱验证码注册 + JWT 双 Token 机制（accessToken + refreshToken）+ Redis 单设备登录**——图形验证码阻止机器批量攻击，邮箱验证码确保身份真实，双 Token 兼顾安全与体验。

## 项目构建全过程

### 第一步：使用 IntelliJ IDEA 创建 Maven 项目

**操作**：在 IDEA 中 New Project → Maven → 设置 JDK 21，创建空 Maven 项目。

**缘由**：
- **Maven** 而非 Gradle：Maven 的 `pom.xml` 声明式依赖管理更直观，国内团队使用更广泛。
- 不使用 Spring Initializr：手动配置 pom.xml 可以精确控制每个依赖的版本，避免 Starter 生成的冗余依赖。

---

### 第二步：配置 pom.xml — 引入 Spring Boot 父工程与依赖

**操作**：写入 [pom.xml](pom.xml)，核心步骤：

1. **设置父 POM 为 `spring-boot-starter-parent:3.5.11`**

   为什么要设置 parent？Spring Boot 的父 POM 预置了：
   - 所有官方 starter 的版本号（无需手动写 `<version>`）
   - Maven 编译插件配置（Java 版本、`-parameters` 编译参数）
   - 资源过滤与打包配置

2. **引入 `spring-boot-starter-web`**

   内嵌 Tomcat 服务器，提供 `@RestController`、`@RequestMapping` 等注解，是整个 Web 层的基础。有了它，启动 `main` 方法就能跑一个 HTTP 服务器。

3. **引入 `mysql-connector-java:8.0.32`**

   MySQL 官方 JDBC 驱动。Spring Boot 通过 `spring.datasource.url` 配置自动加载驱动类。版本 8.0.32 是 MySQL 8.x 系列最稳定的版本之一。

4. **引入 `mybatis-plus-spring-boot3-starter:3.5.12`**

   注意是 `spring-boot3-starter` 而非普通的 `spring-boot-starter`——Spring Boot 3.x 使用了 `jakarta.*` 命名空间（原 `javax.*`），必须用对应的 MyBatis-Plus 分支。同时引入 `mybatis-plus-jsqlparser` 用于分页 SQL 解析。

5. **引入 `spring-boot-starter-data-redis` + `commons-pool2`**

   Redis 客户端默认使用 Lettuce（基于 Netty，异步非阻塞）。`commons-pool2` 提供连接池能力——没有它，Redis 每次操作都要建立新连接，高并发下性能极差。

6. **引入 `spring-boot-starter-security`**

   **这里有一个关键决策**：我们引入了 Spring Security 但不使用它的认证过滤器链。原因：
   - `BCryptPasswordEncoder` 是业界标准的密码哈希算法，自带了盐值，比自己用 MD5/SHA256 安全得多。
   - Spring Security 自动配置的登录表单和 HTTP Basic 认证会干扰我们的 REST API，所以在 `SecurityConfig` 中全部关闭。
   - 为什么不直接用 `spring-security-crypto`？因为它不包含 `SecurityFilterChain` 的自动配置干扰，但同时也无法方便地禁用 CSRF。用 `starter-security` 然后显式关闭更直白。

7. **引入 `jjwt` 三件套**

   - `jjwt-api`：核心接口（compile 作用域）
   - `jjwt-impl`：实现类（runtime 作用域——编译不需要，运行时才加载）
   - `jjwt-jackson`：JSON 序列化用 Jackson（runtime 作用域）

   0.12.x 版本的 API 与旧版不同：`signWith()` 需要 `SecretKey` 对象而非裸字节，`parser()` 链式调用改为 `parser().verifyWith().build().parseSignedClaims()`。

8. **引入 `lombok`**

   项目初期暂时未使用（实体类手写了 getter/setter），但已引入，后续重构可直接用 `@Data` 精简代码。

### 第三步：配置 .gitignore — 版本控制排除策略

**操作**：写了 [.gitignore](.gitignore)，排除以下内容：

- `.DS_Store`：macOS 系统文件

**为什么保留 `!.mvn/wrapper/maven-wrapper.jar`？** Maven Wrapper 允许没有安装 Maven 的机器通过 `mvnw` 脚本自动下载并使用 Maven，`maven-wrapper.jar` 是其核心。`!` 前缀表示不忽略。

### 第四步：初始化数据库 — 编写 init.sql

**操作**：创建 [init.sql](src/main/resources/init.sql)，包含：

```sql
CREATE DATABASE IF NOT EXISTS springboot_zyt
    DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE TABLE IF NOT EXISTS `user` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `email` VARCHAR(100) NOT NULL UNIQUE,
    `password` VARCHAR(255) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**设计要点**：

- **`utf8mb4`**：MySQL 的 `utf8` 只支持最多 3 字节的 BMP 字符，emoji（如 😀）需要 `utf8mb4`（4 字节），不要用 `utf8`。
- **`utf8mb4_unicode_ci`**：排序规则，`ci` 表示大小写不敏感（Case Insensitive），适合用户名搜索。
- **`email` UNIQUE**：数据库层面保证邮箱不重复，防止并发注册时出现同邮箱用户。
- **`password` VARCHAR(255)**：BCrypt 加密后的密文固定 60 字符，255 预留了足够的扩展空间。
- **`id` AUTO_INCREMENT**：自增主键，MyBatis-Plus 的 `IdType.AUTO` 依赖数据库自增。

### 第五步：创建 Spring Boot 启动类

**操作**：创建 [AppStart.java](src/main/java/com/zyt/AppStart.java)。

```java
@SpringBootApplication
@MapperScan("com.zyt.mapper")
public class AppStart {
    public static void main(String[] args) {
        SpringApplication.run(AppStart.class, args);
    }
}
```

**设计要点**：

- **`@SpringBootApplication`**：复合注解，等于 `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`。`@ComponentScan` 默认扫描当前类所在包及其子包，所以所有 Java 代码必须放在 `com.zyt` 及子目录下。
- **`@MapperScan("com.zyt.mapper")`**：告诉 MyBatis-Plus 去哪个包找 Mapper 接口并生成代理实现。不加这个注解，`UserMapper` 注入时会报 `NoSuchBeanDefinitionException`。
- **Spring Boot 的本质**：启动类不是"普通的 Java 类"，它的 `main` 方法会启动内嵌 Tomcat 并注册所有 Bean，之后应用进入监听状态等待 HTTP 请求。

### 第六步：创建实体类 — User

**操作**：创建 [User.java](src/main/java/com/zyt/entity/User.java)。

```java
public class User {
    @TableId(value = "id", type = IdType.AUTO)
    private Integer id;
    @TableField(value = "email")
    private String email;
    private String password;
    // 构造器 + getter/setter
}
```

**设计要点**：

- **`@TableId(type = IdType.AUTO)`**：告诉 MyBatis-Plus `id` 字段由数据库自增生成，`insert` 时不设置该字段值。
- **`@TableField("username")`**：显式映射列名。实际上如果字段名与列名一致（不区分大小写），这个注解是多余的——这里写出来是为了体现"表字段映射"这一概念。
- **`password` 无注解**：因为列名就是 `password`，MyBatis-Plus 会自动进行驼峰映射（`map-underscore-to-camel-case: true`）。
- **为什么不用 Lombok `@Data`？** 教程阶段先手写 getter/setter，让读者明白每个部分的作用。后续重构时替换为 Lombok 可以一次性减少 20 行代码。

### 第七步：创建 Mapper 接口

**操作**：创建 [UserMapper.java](src/main/java/com/zyt/mapper/UserMapper.java)。

```java
public interface UserMapper extends BaseMapper<User> {
}
```

**设计要点**：

- **继承 `BaseMapper<User>`**：MyBatis-Plus 的核心优势。空接口就能获得 `insert`、`deleteById`、`selectById`、`selectList`、`selectPage` 等十余个方法，无需写一行 XML 或注解 SQL。

### 第八步：创建 Service 层

**操作**：创建 [UserService.java](src/main/java/com/zyt/service/UserService.java) 接口 + [UserServiceImpl.java](src/main/java/com/zyt/service/impl/UserServiceImpl.java) 实现类。

Service 层承担了五件事：**图形验证码生成/校验** + **邮箱验证码发送** + **业务逻辑** + **密码加密** + **Redis Token 管理**。

#### 8.1 生成图形验证码

```
generateCaptcha()
  ├─ 调用 CaptchaUtil 生成6位随机字符图片（150×30 PNG）
  ├─ 生成 UUID 作为 captchaId
  ├─ 验证码原文存入 Redis（captcha:{uuid}，2分钟 TTL）
  └─ 返回 captchaId + Base64 图片
```

**图形验证码的防攻击设计**：
- **字符干扰**：4个字符随机旋转±15°，随机粗体/斜体，排除0/O/1/I/l等易混淆字符
- **视觉干扰**：4条随机颜色干扰线 + 30个随机噪点
- **一次性使用**：校验通过后立即从 Redis 删除，防止同一个验证码被重复利用
- **2分钟过期**：即使未通过校验，2分钟后自动失效
- **大小写不敏感**：用户输入不受大小写影响

**为什么用 Java AWT 而不是第三方库？** 简单的图形验证码只需 `BufferedImage` + `Graphics2D` 绘制，JDK 内置的 AWT 完全够用——零额外依赖，打包体积不受影响。

#### 8.2 发送邮箱验证码（需图形验证码）

```
sendCode(email)
  ├─ 校验邮箱格式（正则）
  ├─ 查数据库，该邮箱是否已注册 → 已注册则提示直接登录
  ├─ 防刷：检查 Redis 中是否已有验证码（60秒内不允许重复发送）
  ├─ 生成6位随机数字验证码 → 存入 Redis（5分钟 TTL）
  ├─ 通过163邮箱 SMTP 发送验证码邮件
  └─ 返回发送成功
```

**为什么选163邮箱？** 163邮箱注册即送 SMTP 服务，无需企业认证，个人开发者开箱即用。配置方法：登录 mail.163.com → 设置 → POP3/SMTP/IMAP → 开启SMTP服务 → 获取授权码。

**防刷策略**：同一邮箱60秒内只能发送一次验证码。通过检查 Redis 中验证码的剩余 TTL 实现：如果 TTL > 240秒（即验证码发出不到60秒），拒绝重复发送。

#### 8.2 邮箱验证码注册

```
register(email, password, code)
  ├─ 参数非空校验 + 邮箱格式校验 + 密码长度校验
  ├─ 从 Redis 取出邮箱验证码 → 比对用户输入的 code
  │    ├─ Redis 中无记录 → "验证码已过期或未发送"
  │    └─ 不匹配 → "验证码不正确"
  ├─ 查数据库，邮箱是否已注册
  ├─ BCrypt 加密密码 + 写入用户记录
  ├─ 删除 Redis 中已使用的邮箱验证码（防止重复使用）
  └─ 返回注册成功
```

**为什么注册不用图形验证码？** 注册流程已有邮箱验证码作为身份确认，图形验证码的防机器攻击职责在发送验证码时由60秒防刷机制承担。登录不同——登录是直接比对密码，没有邮箱验证码这一层，所以登录必须保留图形验证码。

**数据库字段 `username` 更名为 `email`**：因为邮箱即用户名，字段名改为 `email` 更语义化。已有数据库需执行：`ALTER TABLE user CHANGE username email VARCHAR(100) NOT NULL UNIQUE;`

**为什么验证码校验完要删除？** 防止同一个验证码被多次使用。一种常见的安全实践。

#### 8.3 登录逻辑 — 双 Token 签发

```
login(username, password)
  ├─ 参数非空校验
  ├─ 查数据库找用户
  ├─ BCrypt 比对密码
  ├─ 检查 Redis 中是否有旧 Token → 有则删除（踢掉旧设备）
  ├─ 生成 accessToken（短期 30 分钟）+ refreshToken（长期 7 天）
  ├─ 两个 Token 分别存入 Redis，TTL 与 JWT 过期时间一致
  └─ 返回双 Token
```

**为什么要双 Token？** 传统的单 Token 方案有一个矛盾：过期时间设短了用户频繁掉线，设长了泄露风险大。双 Token 方案用一个**短期 accessToken**（实际鉴权用，30 分钟过期）搭配一个**长期 refreshToken**（仅用于刷新，7 天过期）解决了这个问题：
- accessToken 即使泄露，30 分钟后自动失效
- refreshToken 只在校验时发送一次（非高频），泄露面极小
- 前端拦截器在 accessToken 过期时自动用 refreshToken 静默换取新的 accessToken，用户完全无感知

**单设备登录的核心就在这里**：Redis 中 key 为 `token:{username}` 存 accessToken，`refresh_token:{username}` 存 refreshToken。同一用户名再次登录时：
1. 读取旧 Token
2. `stringRedisTemplate.delete()` 同时删除两个 Redis key
3. 生成新双 Token 并写入 Redis

此后旧 Token 的持有者访问时，拦截器会从 Redis 中找不到对应的 Token，返回 401。

#### 8.4 登出逻辑

```
logout(token)
  ├─ 解析 Token 获取用户名
  └─ 删除 Redis 中对应的 key
```

登出不需要"让 Token 失效"——JWT 是无状态的，签发后无法撤回。我们能做的是删除 Redis 中的 Token 记录，这样拦截器在下一步校验时会发现 Redis 中没有匹配的 Token，从而拒绝访问。

### 第九步：JWT 工具类 — 双 Token 生成

**操作**：创建 [JwtUtil.java](src/main/java/com/zyt/utils/JwtUtil.java)。

```java
@Component
public class JwtUtil {
    @Value("${jwt.secret}")
    private String secret;
    @Value("${jwt.access-token-expiration}")
    private long accessTokenExpiration;
    @Value("${jwt.refresh-token-expiration}")
    private long refreshTokenExpiration;

    public String generateAccessToken(String username) { ... }
    public String generateRefreshToken(String username) { ... }
    public String getUsernameFromToken(String token) { ... }
    public boolean validateToken(String token) { ... }
}
```

**设计要点**：

- **`@Value` 从配置文件读取密钥和两个过期时间**：accessToken 和 refreshToken 有不同的过期时间，配置独立可灵活调整。
- **`Keys.hmacShaKeyFor()`**：将字符串密钥转为 `SecretKey` 对象。jjwt 0.12.x 要求密钥长度至少 256 位（32 字节）。
- **Token 类型标识**：双 Token 在 JWT claims 中写入 `"type": "access"` 或 `"type": "refresh"`，防止用 refreshToken 当 accessToken 使用。
- **JWT 的 claims（载荷）只存了用户名**：不要存密码、手机号等敏感信息——JWT 的 payload 是 Base64 编码，任何人拿到 Token 都能直接解码查看。签名只防篡改，不防读取。
- **保留 `generateToken()` 兼容方法**：内部调用 `generateAccessToken()`，确保老代码不需要改动。

**双 Token 过期策略**：
- accessToken：30 分钟（`1800000` 毫秒）—— 高频使用，短时效降低泄露风险
- refreshToken：7 天（`604800000` 毫秒）—— 低频使用，长时效避免频繁登录
- Redis 中两个 key 的 TTL 分别与 JWT 过期时间一致，双重保险

### 第十步补充：邮箱发送工具 — 163邮箱 SMTP

**操作**：创建 [EmailUtil.java](src/main/java/com/zyt/utils/EmailUtil.java)。

```java
@Component
public class EmailUtil {
    @Resource
    private JavaMailSender mailSender;

    @Value("${spring.mail.username}")
    private String from;

    public void sendVerificationCode(String to, String code) {
        new Thread(() -> {
            try {
                SimpleMailMessage message = new SimpleMailMessage();
                message.setFrom(from);
                message.setTo(to);
                message.setSubject("Spring Boot Auth — 邮箱验证码");
                message.setText("您的验证码是：" + code + "，5分钟内有效");
                mailSender.send(message);
            } catch (Exception e) {
                e.printStackTrace();  // 发送失败仅打印日志，不影响接口返回
            }
        }).start();
    }
}
```

**设计要点**：

- **异步发送（`new Thread`）**：SMTP 连接163服务器需要数秒（SSL握手+认证+投递），同步发送会阻塞 HTTP 响应，前端 Axios 超时（10s）就会报"网络错误"但邮件实际已发出。异步发送让接口在 Redis 存码后立即返回200，邮件后续在后台投递。
- **使用 `SimpleMailMessage` 而非 `MimeMessage`**：纯文本邮件场景下 `SimpleMailMessage` 更简洁。后续如需 HTML 模板邮件再切换为 `MimeMessageHelper`。
- **`@Value("${spring.mail.username}")` 注入发件人**：163邮箱要求发件人必须是认证邮箱本身。
- **为什么是163邮箱？** 相比 QQ邮箱（授权码+手机绑定流程复杂）、Gmail（国内不可用），163邮箱注册即开 SMTP，个人开发者最友好。

### 如何获取163邮箱授权码

1. 登录 [mail.163.com](https://mail.163.com)
2. 点击 **设置** → **POP3/SMTP/IMAP**
3. 开启 **SMTP 服务**（默认可能关闭）
4. 按提示发送短信验证 → 系统生成**授权码**
5. 将授权码填入 `application.yml` 的 `spring.mail.password`（注意不是邮箱登录密码）

### 第十一步：Spring Security 配置 — 只保留密码加密

**操作**：创建 [SecurityConfig.java](src/main/java/com/zyt/config/SecurityConfig.java)。

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public BCryptPasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth.anyRequest().permitAll());
        return http.build();
    }
}
```

**三个关键禁用的原因**：

1. **禁用 CSRF**：CSRF 保护依赖服务端 Session 中的 CSRF Token。我们的 API 是无状态的（stateless），不存 Session，前端每次请求带 JWT 无需 CSRF Token。禁用它避免所有 POST 请求被 403 拦截。

2. **设为无状态会话**：Spring Security 默认会创建 `HttpSession` 存储安全上下文。设为 `STATELESS` 后，每次请求独立认证——这正是 JWT 的使用模式。

3. **放行所有请求**：认证逻辑由 `LoginInterceptor` 处理，Spring Security 的过滤器链不再拦截任何请求。如果不放行，Spring Security 默认会要求所有请求通过认证。

**引入 Spring Security 的唯一目的**：获得 `BCryptPasswordEncoder` 这个 Bean。这样做比直接引入 `spring-security-crypto` 更"标准"，因为大项目中迟早会用到 Spring Security 的其他功能（如方法级权限控制）。

### 第十二步：登录拦截器 — 自定义 Token 校验

**操作**：创建 [LoginInterceptor.java](src/main/java/com/zyt/config/LoginInterceptor.java)。

这是本项目鉴权体系的**核心**，校验链路如下：

```
请求到达
  ├─ 1. 从 Authorization 请求头提取 Token
  │    └─ 为空 → 401 "未携带 Token"
  ├─ 2. JWT 自校验（签名 + 过期）
  │    └─ 无效/过期 → 401 "Token 无效或已过期"
  ├─ 3. 从 JWT 中解析出用户名
  ├─ 4. 用用户名查 Redis，获取当前有效 Token
  │    └─ Redis 中无记录 或 不匹配 → 401 "Token 已失效（被新登录踢掉或已登出）"
  └─ 放行，进入 Controller
```

**为什么用 `HandlerInterceptor` 而非 Spring Security Filter？**
- `HandlerInterceptor` 是 Spring MVC 层面的组件，能拿到已解析的 `HandlerMethod`，可以做更精细的控制。
- 配置更简单——在 `WebConfig` 中注册拦截器即可，不需要理解 Spring Security 的过滤器链顺序。
- 由于已经在 `SecurityConfig` 中关闭了 Spring Security 的所有保护，用拦截器是更直白的写法。

**关于 Token 存储位置**：
- 客户端发送 Token 用 `Authorization` 请求头（而非自定义头如 `X-Token`），这是 HTTP 标准做法，便于后续接入 Swagger、Apifox 等工具时直接使用标准认证配置。
- Redis key 设计为 `token:{username}`：前缀区分命名空间，避免与其他业务 key 冲突。

**拦截器返回 `false` 时**：请求不会进入 Controller，直接返回 401 状态码 + JSON 错误体。

### 第十三步：Web 配置 — 注册拦截器

**操作**：创建 [WebConfig.java](src/main/java/com/zyt/config/WebConfig.java)。

```java
@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(loginInterceptor)
                .addPathPatterns("/**")              // 拦截所有请求
                .excludePathPatterns("/user/login", "/user/register", "/user/refresh");  // 放行
    }
}
```

**放行 `/user/login` 和 `/user/register`**：这两个接口本身就是用来获取 Token 的，如果拦截它们就成了"先有鸡还是先有蛋"的死循环。`/user/refresh` 也放行——刷新 Token 时 accessToken 已经过期，不能用它来通过拦截器校验，而是用 refreshToken 在 Controller 层直接验证。

**`excludePathPatterns` 只放行了 `/user/**` 下的三个特定路径**，不是放行整个 `/user/**`。这意味着 `/user/logout` 和 `/user/info` 仍会被拦截。

#### 数据源配置


**连接串各参数的作用**：
- `useSSL=false`：本地开发无需 SSL 加密，避免证书问题
- `serverTimezone=Asia/Shanghai`：指定时区，否则 MySQL 8.x 驱动会报 `The server time zone value 'XXX' is unrecognized` 错误

- `allowPublicKeyRetrieval=true`：允许客户端从服务器获取公钥，mysql 8.0 默认的 `caching_sha2_password` 认证需要这个

#### Redis 连接池配置

```yaml
spring:
  data:
    redis:
      host: localhost
      port: 6379
      lettuce:
        pool:
          max-active: 8     # 最大活跃连接数
          max-idle: 8       # 最大空闲连接数
          min-idle: 0       # 最小空闲连接数
          max-wait: -1ms    # 获取连接最大等待时间，-1 为无限等待
```

Lettuce 是 Spring Boot 2.x+ 默认的 Redis 客户端，替换了旧版 Jedis。它基于 Netty，单连接就能并发处理多个命令（Jedis 需要连接池才能并发）。连接池参数适用于高并发场景。

```yaml

    map-underscore-to-camel-case: true                      # 下划线转驼峰
  type-aliases-package: com.zyt.entity                       # 实体类扫描包
```

**SQL 日志输出到标准输出**：开发阶段非常有价值——能看到每一条实际执行的 SQL 和参数，便于调试。生产环境应关闭。

#### JWT 配置

```yaml
jwt:
  secret: zyt-jwt-secret-key-2026-springboot-login-system
  access-token-expiration: 1800000     # 30 分钟
  refresh-token-expiration: 604800000  # 7 天
```

**`secret` 为什么这么长？** jjwt 0.12.x 要求 HMAC-SHA256 密钥至少 256 bits（32 字节）。这里的字符串刚好超过 32 字符，满足要求。

**两个过期时间分别控制**：accessToken 30 分钟保证安全性（即使泄露影响窗口小），refreshToken 7 天保证用户体验（一周内不用重新登录）。这两个值可根据业务需求独立调整。

#### 邮箱配置（163 SMTP）

```yaml
spring:
  mail:
    host: smtp.163.com
    port: 465                       # SSL 加密端口
    username: your_email@163.com    # 替换为你的163邮箱
    password: your_smtp_auth_code   # 替换为SMTP授权码（非登录密码）
    protocol: smtps                 # SMTPS 协议（SSL加密）
    properties:
      mail:
        smtp:
          auth: true                # 启用认证
          ssl:
            enable: true            # 启用 SSL
          starttls:
            enable: true
            required: true
```

**为什么用465端口（SSL）而非25/587？** 25端口是SMTP默认端口但不加密，587端口需要STARTTLS升级，465端口从连接开始就是SSL加密——163邮箱官方推荐使用465端口（标注为"smtps"协议）。

**`password` 不是邮箱登录密码！** 必须使用163邮箱的 SMTP 授权码（在设置中单独生成），直接使用登录密码会导致认证失败。

# 第二部分：前端（Vue 3 + Element Plus）

## 前言：为什么做前后端分离

在本教程第一部分中，后端已完成了完整的 RESTful API（注册、登录、登出、鉴权）。但我们没有在后端工程里直接写前端页面——Spring Boot 的 `static` 目录虽然可以托管静态文件，但在实际开发中存在几个问题：

1. **前后端耦合**：改前端样式也要重新编译打包整个后端项目
2. **开发体验差**：改一行 CSS 要等 Maven 重新构建，没有热更新
3. **团队协作困难**：前端和后端开发者在同一仓库里互相等待
4. **部署不灵活**：前端静态文件适合放 CDN 或 Nginx，不适合放在 Java 进程里

所以我们将前端独立为一个 **Vue 3 + Vite + Element Plus** 工程，与后端 **完全分离**。

---

## 第十八步：初始化 Vue 3 + Vite 项目

**操作**：在项目根目录执行：

```bash
npm create vite@latest frontend -- --template vue
cd frontend
npm install
```

**为什么用 Vite 而不是 Vue CLI？**
- Vue CLI 基于 Webpack，项目大了之后开发服务器启动慢、热更新慢
- Vite 基于原生 ES Module，开发服务器秒启动，HMR 几乎瞬间生效
- Vite 8.x 使用 Rolldown（Rust 编写的打包器）做生产构建，速度远超 Webpack

**为什么 `frontend` 放在后端项目根目录下？**
- 同一个 Git 仓库便于管理，一套 `git clone` 拿到前后端全部代码
- 但它们是两个独立工程，各有各的 `package.json` 和 `pom.xml`，互不依赖

**操作**：进入 `frontend/` 目录，安装运行时依赖和开发依赖。

## 第二十步：配置 Vite — 代理与按需导入

**操作**：编写 [vite.config.js](frontend/vite.config.js)。

```javascript
import { defineConfig } from 'vite'
import vue from '@vitejs/plugin-vue'
import AutoImport from 'unplugin-auto-import/vite'
import Components from 'unplugin-vue-components/vite'
import { ElementPlusResolver } from 'unplugin-vue-components/resolvers'
import path from 'path'

export default defineConfig({
  plugins: [
    vue(),
    AutoImport({ resolvers: [ElementPlusResolver()] }),
    Components({ resolvers: [ElementPlusResolver()] }),
  ],
  resolve: {
    alias: { '@': path.resolve(__dirname, 'src') },
  },
  server: {
    port: 3000,
    proxy: {
      '/user': {
        target: 'http://localhost:8080',
        changeOrigin: true,
      },
    },
  },
})
```

**三个关键配置的缘由**：

1. **`@` 路径别名**：`import { login } from '@/api/user'` 替代 `../../../api/user`。Vite 编译时会将 `@` 解析为 `src/` 目录的绝对路径。

2. **`server.proxy` 代理配置**：前端运行在 `localhost:3000`，后端在 `localhost:8080`。如果不做代理，浏览器发起 `POST http://localhost:3000/user/login` 会 404。配置代理后，Vite 开发服务器将所有 `/user` 开头的请求转发到 `http://localhost:8080`，对浏览器来说它以为自己在跟 3000 端口通信——这就是开发阶段的**跨域解决方案**。

3. **`ElementPlusResolver`**：告诉 `unplugin-vue-components` 去 Element Plus 包里找组件。它的工作流程是：
   - 扫描 `.vue` 文件中的模板标签
   - 发现 `<el-button>` → 自动添加 `import { ElButton } from 'element-plus'` 
   - 同时自动引入对应的 CSS 样式文件

   > 如果不用这个 resolver，每次用新组件都要手动 `import` 并在 `components` 中注册。

## 第二十一步：Token 管理工具 — 双 Token 存储

**操作**：创建 [token.js](frontend/src/utils/token.js)。

```javascript
const ACCESS_TOKEN_KEY = 'zyt_access_token'
const REFRESH_TOKEN_KEY = 'zyt_refresh_token'

export function getToken() { return localStorage.getItem(ACCESS_TOKEN_KEY) }
export function getRefreshToken() { return localStorage.getItem(REFRESH_TOKEN_KEY) }

export function setTokens(accessToken, refreshToken) {
  localStorage.setItem(ACCESS_TOKEN_KEY, accessToken)
  localStorage.setItem(REFRESH_TOKEN_KEY, refreshToken)
}
export function setToken(token) { localStorage.setItem(ACCESS_TOKEN_KEY, token) }
export function removeToken() {
  localStorage.removeItem(ACCESS_TOKEN_KEY)
  localStorage.removeItem(REFRESH_TOKEN_KEY)
}
```

**为什么用两个 key 分别存储？**
- accessToken 是高频读写的（每次请求都要从 localStorage 读取），单独存避免解析 JSON 的开销
- refreshToken 只在自动刷新时读取一次，与 accessToken 生命周期解耦
- `removeToken()` 同时清除两个 Token，保证登出时完全清理

**为什么用 localStorage 存 Token？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| Cookie | 自动随请求发送 | 有 CSRF 风险，需要额外配置 SameSite |
| sessionStorage | 关闭标签页自动清除 | 用户刷新页面就丢失，不实用 |
| **localStorage** | 持久化存储，跨标签页共享 | XSS 可读取（但 Vue 默认转义输出，风险可控） |

我们选择 `localStorage`：前端 SPA 中 Vue 会自动转义所有模板变量，XSS 攻击面很小。存储键名加了前缀 `zyt_` 以避免与其他项目冲突。

## 第二十二步：Axios 请求层 — 拦截器机制

**操作**：创建 [api/index.js](frontend/src/api/index.js)，这是前端与后端通信的**核心枢纽**。

```javascript
import axios from 'axios'
import { getToken, removeToken } from '@/utils/token'
import { ElMessage } from 'element-plus'

const request = axios.create({
  baseURL: '/',
  timeout: 10000,
})
```

### 请求拦截器 — 自动带 Token

```javascript
request.interceptors.request.use(
  (config) => {
    const token = getToken()
    if (token) {
      config.headers.Authorization = token
    }
    return config
  },
  (error) => Promise.reject(error),
)
```

**为什么要在拦截器里统一加 Token 而不是在每个请求里手动加？**
- 集中管理：将来如果改用 `Bearer xxx` 格式，只需改一处
- 不会漏加：开发者调用 API 时不需要记得传 Token
- 自动判断：有 Token 就加，没有就不加（注册和登录时还没有 Token）

### 响应拦截器 — 统一错误处理 + 自动无感刷新

```javascript
let isRefreshing = false
let pendingRequests = []

request.interceptors.response.use(
  (response) => {
    const res = response.data
    if (res.code !== 200) {
      ElMessage.error(res.msg || '请求失败')
      return Promise.reject(new Error(res.msg))
    }
    return res
  },
  async (error) => {
    const { config, response } = error
    // 401 时尝试自动刷新 Token
    if (response && response.status === 401 && !config._retry) {
      const refreshToken = getRefreshToken()
      if (!refreshToken) { /* 无 refreshToken，直接踢到登录页 */ }
      // 如果正在刷新中，将请求排队等待
      if (isRefreshing) {
        return new Promise((resolve) => {
          addPendingRequest((newToken) => {
            config.headers.Authorization = newToken
            resolve(request(config))  // 用新 Token 重试
          })
        })
      }
      isRefreshing = true
      try {
        const res = await axios.post('/user/refresh', { refreshToken })
        setToken(res.data.data.accessToken)
        onRefreshed(res.data.data.accessToken)  // 重放所有排队的请求
        config.headers.Authorization = newAccessToken
        return request(config)  // 重试当前请求
      } catch {
        removeToken()
        window.location.href = '/login'
      } finally { isRefreshing = false }
    }
    ...
  },
)
```

**自动刷新机制的完整流程**：

```
请求 → 后端返回 401
  ├─ 无 refreshToken → 直接踢到登录页
  ├─ 有 refreshToken，且当前没有在刷新中
  │    ├─ 标记 isRefreshing = true
  │    ├─ 调用 /user/refresh 获取新 accessToken
  │    ├─ 更新 localStorage 中的 accessToken
  │    ├─ 重放所有排队的请求（用新 Token）
  │    └─ 重试当前请求
  └─ 有 refreshToken，但已经在刷新中
       └─ 将当前请求加入 pendingRequests 队列，等刷新完成后自动重试
```

**为什么需要请求队列（`pendingRequests`）？** 假设页面同时发了 3 个请求，它们都因 accessToken 过期而返回 401。如果不做排队，3 个请求会同时尝试刷新 Token——导致多次刷新、浪费资源、可能出现竞态条件。用 `isRefreshing` 标志 + 队列，确保同一时间只有一个刷新请求，其他请求等待共享结果。

**`config._retry` 的作用**：防止重试后仍然 401 导致死循环。已重试过的请求不再触发刷新逻辑。

**两层错误处理的设计逻辑**：

1. **HTTP 200 但业务码非 200**（如"用户名已存在"、"密码错误"）：后端返回 `{ code: 400, msg: "密码错误" }`，拦截器直接弹出错误提示。

2. **HTTP 401**（accessToken 过期）：自动用 refreshToken 无感刷新，用户完全无感知。只有 refreshToken 也过期时才跳转登录页。

3. **网络错误**（后端宕机、超时等）：提示通用错误信息。

```javascript
  {
    path: '/home',
    name: 'Home',
    component: () => import('@/views/Home.vue'),
    meta: { requiresAuth: true },
  },

```

### 懒加载

`() => import('@/views/Login.vue')` 而非 `import Login from '@/views/Login.vue'`：
- 静态 `import` 会把所有页面打包到一个 JS 文件，首屏加载慢
- 动态 `import()` 让 Vite 把每个页面拆成独立的 chunk，只有访问时才加载
- 三个页面 → 三个独立 JS 文件，首屏只下载当前页面需要的那一个

### 导航守卫 — 前端鉴权

```javascript
router.beforeEach((to, from, next) => {
  const token = getToken()
  if (to.meta.requiresAuth && !token) {
    next('/login')        // 未登录 → 去登录
  } else if (to.meta.guest && token) {
    next('/home')          // 已登录 → 不让你看登录页
  } else {
    next()                 // 放行
  }
})
```

**三层判断**：

| 场景 | Token | 目标页 | 结果 |
|------|-------|--------|------|
| 未登录访问首页 | 无 | `/home` (requiresAuth) | → 跳到 `/login` |
| 已登录访问登录页 | 有 | `/login` (guest) | → 跳到 `/home` |
| 正常访问 | 任意 | 匹配 | 放行 |

> **注意**：前端路由守卫只是**用户体验优化**（避免用户看到空白页），真正的安全防护在后端 `LoginInterceptor`。攻击者可以直接用 curl 绕过前端，但绕不过后端的拦截器。

## 开发中的常见问题与解决方案

### Q1：`npm run dev` 后页面空白

**原因**：Vite 构建缓存问题或 Vite 版本不兼容。

**解决**：删除 `node_modules` 和 `package-lock.json`，重新 `npm install`。

### Q2：登录后立刻被跳回登录页

**原因**：路由守卫检测不到 Token。检查：
1. 浏览器 DevTools → Application → Local Storage 中是否有 `zyt_access_token` 和 `zyt_refresh_token`
2. 后端登录接口是否真的返回了双 Token（检查 Network 标签的响应体）

### Q3：请求发到了 3000 端口而不是 8080

**原因**：代理只对开发服务器生效。检查：
1. `axios` 的 `baseURL` 设置为 `/` 而不是 `http://localhost:8080`
2. Vite dev server 必须通过 `npm run dev` 启动，直接打开 HTML 文件不行

### Q4：Token 自动刷新机制是如何工作的？

1. accessToken 过期（30 分钟）→ 后端返回 401
2. Axios 响应拦截器捕获 401，用 refreshToken 自动请求 `/user/refresh`
3. 获得新 accessToken，更新 localStorage，用新 Token 重试原请求
4. 如果同时有多个请求返回 401，第一个会触发刷新，其余排队等待
5. 只有当 refreshToken 也过期（7 天）时，用户才需要重新登录

### Q5：页面样式异常，按钮没有样式

**原因**：Element Plus CSS 未引入。

## 项目架构分层（全栈）

```
浏览器 → Vue Router → Axios ──HTTP──→ LoginInterceptor → Controller → Service → Mapper → Database
                │                          │                    ↕          ↕
           localStorage              JWT + Redis             JwtUtil     Redis
        (accessToken +                   │
         refreshToken)            token:{username}
                                  refresh_token:{username}
```

**后端每层的职责边界**：
- **Controller**：解析 HTTP 参数，调用 Service，返回 JSON。不应包含任何业务逻辑。
- **Service**：业务逻辑 + 事务管理。本项目未加 `@Transactional`，因为都是单表操作。
- **Mapper**：数据访问，MyBatis-Plus 自动生成，不需要手写。
- **Config**：拦截器 / 安全配置 / 分页插件，相当于项目的"基础设施"。
- **Utils**：纯工具类，JWT 和统一响应，无业务依赖。

**前端每层的职责边界**：
- **Views**：页面组件，专注 UI 交互和表单校验，调用 API 层获取数据
- **API**：Axios 实例 + 请求/响应拦截器（含自动刷新） + 接口方法封装
- **Router**：路由配置 + `beforeEach` 导航守卫，实现前端鉴权
- **Utils**：双 Token 存取工具，桥接 API 层和 localStorage

## 关键设计决策速查表

| 决策 | 选择 | 理由 |
|------|------|------|
| 前后端架构 | 完全分离 | 前端独立工程，Vite 代理解决跨域，后端零前端代码 |
| Token 策略 | 双 Token（access + refresh） | 短期 accessToken 降低泄露风险，长期 refreshToken 无需频繁登录 |
| 无感刷新 | Axios 拦截器 + 请求队列 | 401 时自动用 refreshToken 刷新，多请求并发时排队防止重复刷新 |
| 认证方式 | 自定义拦截器 + JWT | 比 Spring Security 更简单直白，适合 RESTful API |
| 密码存储 | BCrypt | 自带随机盐，每密文不同，抗彩虹表 |
| 单设备登录 | Redis 存双 Token | 利用 Redis TTL 天然过期，读写快，支持同时失效两个 Token |
| 登出实现 | 删 Redis key | JWT 无法主动失效，只能通过服务端额外存储来实现撤销 |
| CSRF | 显式禁用 | JWT 认证下 CSRF 不存在（不依赖 Cookie 传认证信息） |
| 会话管理 | 无状态 | RESTful API 的标准做法，服务器不存用户 Session |
| ORM | MyBatis-Plus | MyBatis 的灵活性 + JPA 式单表 CRUD 免编写 |
| 数据库字符集 | utf8mb4 | 支持 emoji，避免 utf8(3字节) 的坑 |
| 前端 Token 存储 | localStorage | 简单直接，SPA 应用中无 XSS 风险（Vue 自动转义） |
| UI 组件 | Element Plus 按需导入 | `unplugin-vue-components` 自动注册，减小打包体积 |
| 登录页布局 | 双栏左右布局 + 选项卡切换 | 左侧表单区 + 右侧授权/权益信息区，移动端自动上下布局 |
| 微信登录 | 预留占位 UI | 选项卡和图标入口已就绪，后续接入 OAuth 2.0 替换占位即可 |
| 图形验证码 | Java AWT 绘制 + Redis 2min TTL | JDK 内置，干扰线+噪点+字符旋转，一次性使用 |
| 邮箱验证码 | 163邮箱 SMTP + Redis 5min TTL | 个人开发者免费可用，Redis 自然过期无需手动清理 |
| 防刷策略 | 图形验证码 + Redis TTL 检查（60秒间隔） | 双重防护：图形验证码阻止机器，60秒限制阻止脚本 |
| 颜色主题 | 绿色 `#31c27c` | 清新安全感的品牌色，与 BCrypt/Token 安全机制传达一致 |
| 开发代理 | Vite proxy | 开发时 `/user` 请求代理到 8080，避免跨域问题 |

## 启动与测试步骤

1. **启动 MySQL**，执行 `init.sql` 创建数据库和表
2. **启动 Redis**，默认端口 6379
3. **启动后端**：运行 `AppStart.main()`，监听 8080 端口
4. **启动前端**：进入 `frontend/` 目录，运行 `npm install && npm run dev`，监听 3000 端口
5. 打开浏览器访问 `http://localhost:3000`
6. 看到双栏登录页 → 默认"密码登录"选项卡 → 点击底部「注册账号」
7. 进入双栏注册页 → 输入邮箱 → 点击「发送验证码」→ 查看邮箱获取验证码
8. 输入验证码 + 密码 → 点击注册 → 注册成功跳转登录页
9. 输入邮箱+密码登录 → 进入首页看到「鉴权通过」（含 Token 刷新策略显示）
10. 切换到"微信登录"选项卡体验预留的扫码界面 → 返回密码登录
11. 点击「退出登录」清理 Token 并返回登录页

也可以用 [APITest.http](APITest.http) 单独测试后端 API，验证邮箱验证码发送和注册流程。

# Spring Boot + Vue3 用户认证系统 — 构建教程

## 项目概述

本项目是一个 **前后端分离** 的用户注册、登录、鉴权系统：

- **后端**：Spring Boot 3.5.11 + MyBatis-Plus + MySQL + Redis + JavaMail（163邮箱）
- **前端**：Vue 3 + Vite + Element Plus + Vue Router + Axios

核心特色是 **图形验证码防刷 + 邮箱验证码注册 + JWT 双 Token 机制（accessToken + refreshToken）+ Redis 单设备登录 + 暴力破解防护 + 微信 OAuth 扫码登录（真实扫码+邮箱绑定）**——图形验证码阻止机器批量攻击，邮箱验证码确保身份真实，双 Token 兼顾安全与体验，微信扫码登录提升用户体验。

## 项目构建全过程

### 第一步：使用 IntelliJ IDEA 创建 Maven 项目

**操作**：在 IDEA 中 New Project → Maven → 设置 JDK 21，创建空 Maven 项目。

**缘由**：
- **Maven** 而非 Gradle：Maven 的 `pom.xml` 声明式依赖管理更直观，国内团队使用更广泛。
- 不使用 Spring Initializr：手动配置 pom.xml 可以精确控制每个依赖的版本，避免 Starter 生成的冗余依赖。

---

### 第二步：配置 pom.xml — 引入 Spring Boot 父工程与依赖

**操作**：写入 [pom.xml](pom.xml)，核心步骤：

1. **设置父 POM 为 `spring-boot-starter-parent:3.5.11`**

   为什么要设置 parent？Spring Boot 的父 POM 预置了：
   - 所有官方 starter 的版本号（无需手动写 `<version>`）
   - Maven 编译插件配置（Java 版本、`-parameters` 编译参数）
   - 资源过滤与打包配置

2. **引入 `spring-boot-starter-web`**

   内嵌 Tomcat 服务器，提供 `@RestController`、`@RequestMapping` 等注解，是整个 Web 层的基础。有了它，启动 `main` 方法就能跑一个 HTTP 服务器。

3. **引入 `mysql-connector-java:8.0.32`**

   MySQL 官方 JDBC 驱动。Spring Boot 通过 `spring.datasource.url` 配置自动加载驱动类。版本 8.0.32 是 MySQL 8.x 系列最稳定的版本之一。

4. **引入 `mybatis-plus-spring-boot3-starter:3.5.12`**

   注意是 `spring-boot3-starter` 而非普通的 `spring-boot-starter`——Spring Boot 3.x 使用了 `jakarta.*` 命名空间（原 `javax.*`），必须用对应的 MyBatis-Plus 分支。同时引入 `mybatis-plus-jsqlparser` 用于分页 SQL 解析。

5. **引入 `spring-boot-starter-data-redis` + `commons-pool2`**

   Redis 客户端默认使用 Lettuce（基于 Netty，异步非阻塞）。`commons-pool2` 提供连接池能力——没有它，Redis 每次操作都要建立新连接，高并发下性能极差。

6. **引入 `spring-boot-starter-security`**

   **这里有一个关键决策**：我们引入了 Spring Security 但不使用它的认证过滤器链。原因：
   - `BCryptPasswordEncoder` 是业界标准的密码哈希算法，自带了盐值，比自己用 MD5/SHA256 安全得多。
   - Spring Security 自动配置的登录表单和 HTTP Basic 认证会干扰我们的 REST API，所以在 `SecurityConfig` 中全部关闭。
   - 为什么不直接用 `spring-security-crypto`？因为它不包含 `SecurityFilterChain` 的自动配置干扰，但同时也无法方便地禁用 CSRF。用 `starter-security` 然后显式关闭更直白。

7. **引入 `jjwt` 三件套**

   - `jjwt-api`：核心接口（compile 作用域）
   - `jjwt-impl`：实现类（runtime 作用域——编译不需要，运行时才加载）
   - `jjwt-jackson`：JSON 序列化用 Jackson（runtime 作用域）

   0.12.x 版本的 API 与旧版不同：`signWith()` 需要 `SecretKey` 对象而非裸字节，`parser()` 链式调用改为 `parser().verifyWith().build().parseSignedClaims()`。

8. **引入 `lombok`**

   项目初期暂时未使用（实体类手写了 getter/setter），但已引入，后续重构可直接用 `@Data` 精简代码。

**操作**：写了 [.gitignore](.gitignore)，排除以下内容：

### 第四步：初始化数据库 — 编写 init.sql

**操作**：创建 [init.sql](src/main/resources/init.sql)，包含：

```sql
CREATE DATABASE IF NOT EXISTS springboot_zyt
    DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;

CREATE TABLE IF NOT EXISTS `user` (
    `id` INT AUTO_INCREMENT PRIMARY KEY,
    `email` VARCHAR(100) NOT NULL UNIQUE,
    `password` VARCHAR(255) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

**设计要点**：

- **`utf8mb4`**：MySQL 的 `utf8` 只支持最多 3 字节的 BMP 字符，emoji（如 😀）需要 `utf8mb4`（4 字节），不要用 `utf8`。
- **`utf8mb4_unicode_ci`**：排序规则，`ci` 表示大小写不敏感（Case Insensitive），适合用户名搜索。
- **`email` UNIQUE**：数据库层面保证邮箱不重复，防止并发注册时出现同邮箱用户。
- **`password` VARCHAR(255)**：BCrypt 加密后的密文固定 60 字符，255 预留了足够的扩展空间。
- **`id` AUTO_INCREMENT**：自增主键，MyBatis-Plus 的 `IdType.AUTO` 依赖数据库自增。

### 第五步：创建 Spring Boot 启动类

**操作**：创建 [AppStart.java](src/main/java/com/zyt/AppStart.java)。

```java
@SpringBootApplication
@MapperScan("com.zyt.mapper")
public class AppStart {
    public static void main(String[] args) {
        SpringApplication.run(AppStart.class, args);
    }
}
```

**设计要点**：

- **`@SpringBootApplication`**：复合注解，等于 `@Configuration` + `@EnableAutoConfiguration` + `@ComponentScan`。`@ComponentScan` 默认扫描当前类所在包及其子包，所以所有 Java 代码必须放在 `com.zyt` 及子目录下。
- **`@MapperScan("com.zyt.mapper")`**：告诉 MyBatis-Plus 去哪个包找 Mapper 接口并生成代理实现。不加这个注解，`UserMapper` 注入时会报 `NoSuchBeanDefinitionException`。
- **Spring Boot 的本质**：启动类不是"普通的 Java 类"，它的 `main` 方法会启动内嵌 Tomcat 并注册所有 Bean，之后应用进入监听状态等待 HTTP 请求。

### 第六步：创建实体类 — User

**操作**：创建 [User.java](src/main/java/com/zyt/entity/User.java)。

```java
public class User {
    @TableId(value = "id", type = IdType.AUTO)
    private Integer id;
    @TableField(value = "email")
    private String email;
    private String password;
    // 构造器 + getter/setter
}
```

**设计要点**：

- **`@TableId(type = IdType.AUTO)`**：告诉 MyBatis-Plus `id` 字段由数据库自增生成，`insert` 时不设置该字段值。
- **`@TableField("username")`**：显式映射列名。实际上如果字段名与列名一致（不区分大小写），这个注解是多余的——这里写出来是为了体现"表字段映射"这一概念。
- **`password` 无注解**：因为列名就是 `password`，MyBatis-Plus 会自动进行驼峰映射（`map-underscore-to-camel-case: true`）。
- **为什么不用 Lombok `@Data`？** 教程阶段先手写 getter/setter，让读者明白每个部分的作用。后续重构时替换为 Lombok 可以一次性减少 20 行代码。

### 第七步：创建 Mapper 接口

**操作**：创建 [UserMapper.java](src/main/java/com/zyt/mapper/UserMapper.java)。

```java
public interface UserMapper extends BaseMapper<User> {
}
```

**设计要点**：

- **继承 `BaseMapper<User>`**：MyBatis-Plus 的核心优势。空接口就能获得 `insert`、`deleteById`、`selectById`、`selectList`、`selectPage` 等十余个方法，无需写一行 XML 或注解 SQL。

### 第八步：创建 Service 层

**操作**：创建 [UserService.java](src/main/java/com/zyt/service/UserService.java) 接口 + [UserServiceImpl.java](src/main/java/com/zyt/service/impl/UserServiceImpl.java) 实现类。

Service 层承担了五件事：**图形验证码生成/校验** + **邮箱验证码发送** + **业务逻辑** + **密码加密** + **Redis Token 管理**。

#### 8.1 生成图形验证码

```
generateCaptcha()
  ├─ 调用 CaptchaUtil 生成6位随机字符图片（150×30 PNG）
  ├─ 生成 UUID 作为 captchaId
  ├─ 验证码原文存入 Redis（captcha:{uuid}，2分钟 TTL）
  └─ 返回 captchaId + Base64 图片
```

**图形验证码的防攻击设计**：
- **字符干扰**：4个字符随机旋转±15°，随机粗体/斜体，排除0/O/1/I/l等易混淆字符
- **视觉干扰**：4条随机颜色干扰线 + 30个随机噪点
- **一次性使用**：校验通过后立即从 Redis 删除，防止同一个验证码被重复利用
- **2分钟过期**：即使未通过校验，2分钟后自动失效
- **大小写不敏感**：用户输入不受大小写影响

**为什么用 Java AWT 而不是第三方库？** 简单的图形验证码只需 `BufferedImage` + `Graphics2D` 绘制，JDK 内置的 AWT 完全够用——零额外依赖，打包体积不受影响。

#### 8.2 发送邮箱验证码（需图形验证码）

```
sendCode(email)
  ├─ 校验邮箱格式（正则）
  ├─ 查数据库，该邮箱是否已注册 → 已注册则提示直接登录
  ├─ 防刷：检查 Redis 中是否已有验证码（60秒内不允许重复发送）
  ├─ 生成6位随机数字验证码 → 存入 Redis（5分钟 TTL）
  ├─ 通过163邮箱 SMTP 发送验证码邮件
  └─ 返回发送成功
```

**为什么选163邮箱？** 163邮箱注册即送 SMTP 服务，无需企业认证，个人开发者开箱即用。配置方法：登录 mail.163.com → 设置 → POP3/SMTP/IMAP → 开启SMTP服务 → 获取授权码。

**防刷策略**：同一邮箱60秒内只能发送一次验证码。通过检查 Redis 中验证码的剩余 TTL 实现：如果 TTL > 240秒（即验证码发出不到60秒），拒绝重复发送。

#### 8.2 邮箱验证码注册

```
register(email, password, code)
  ├─ 参数非空校验 + 邮箱格式校验 + 密码长度校验
  ├─ 从 Redis 取出邮箱验证码 → 比对用户输入的 code
  │    ├─ Redis 中无记录 → "验证码已过期或未发送"
  │    └─ 不匹配 → "验证码不正确"
  ├─ 查数据库，邮箱是否已注册
  ├─ BCrypt 加密密码 + 写入用户记录
  ├─ 删除 Redis 中已使用的邮箱验证码（防止重复使用）
  └─ 返回注册成功
```

**为什么注册不用图形验证码？** 注册流程已有邮箱验证码作为身份确认，图形验证码的防机器攻击职责在发送验证码时由60秒防刷机制承担。登录不同——登录是直接比对密码，没有邮箱验证码这一层，所以登录必须保留图形验证码。

**数据库字段 `username` 更名为 `email`**：因为邮箱即用户名，字段名改为 `email` 更语义化。已有数据库需执行：`ALTER TABLE user CHANGE username email VARCHAR(100) NOT NULL UNIQUE;`

**为什么验证码校验完要删除？** 防止同一个验证码被多次使用。一种常见的安全实践。

#### 8.3 登录逻辑 — 双 Token 签发 + 暴力破解防护

```
login(email, password, captchaId, captchaCode)
  ├─ 参数非空校验 + 图形验证码校验（一次性使用，通过后删除）
  ├─ 防暴力破解：检查 Redis 登录失败计数
  │    └─ 连续失败 ≥5 次 → 429 "登录失败次数过多，请15分钟后再试"
  ├─ 查数据库找用户
  ├─ BCrypt 比对密码
  │    └─ 用户不存在 或 密码错误 → INCR 失败计数 + 统一返回"邮箱或密码错误"
  ├─ 登录成功 → 删除 Redis 失败计数
  ├─ 检查 Redis 中是否有旧 Token → 有则删除（踢掉旧设备）
  ├─ 生成 accessToken（短期 30 分钟，type=access）+ refreshToken（长期 7 天，type=refresh）
  ├─ 两个 Token 分别存入 Redis，TTL 与 JWT 过期时间一致
  └─ 返回双 Token
```

**为什么要双 Token？** 传统的单 Token 方案有一个矛盾：过期时间设短了用户频繁掉线，设长了泄露风险大。双 Token 方案用一个**短期 accessToken**（实际鉴权用，30 分钟过期）搭配一个**长期 refreshToken**（仅用于刷新，7 天过期）解决了这个问题：
- accessToken 即使泄露，30 分钟后自动失效
- refreshToken 只在校验时发送一次（非高频），泄露面极小
- JWT claims 中包含 `type` 字段（`"access"` 或 `"refresh"`），`/user/refresh` 接口强制校验 `type=refresh`，防止攻击者用 accessToken 无限续期（P0 安全修复）
- 前端拦截器在 accessToken 过期时自动用 refreshToken 静默换取新的 accessToken，用户完全无感知

**暴力破解防护**：同一邮箱连续 5 次登录失败后锁定 15 分钟，返回 HTTP 429。Redis `INCR` 原子计数，TTL 仅在首次失败时设置（防止攻击者通过控制失败间隔重置计时器）。登录成功则清除失败计数。所有失败统一返回「邮箱或密码错误」，防止用户枚举攻击。

**单设备登录的核心就在这里**：Redis 中 key 为 `token:{email}` 存 accessToken，`refresh_token:{email}` 存 refreshToken。同一用户名再次登录时：
1. 读取旧 Token
2. `stringRedisTemplate.delete()` 同时删除两个 Redis key
3. 生成新双 Token 并写入 Redis

此后旧 Token 的持有者访问时，拦截器会从 Redis 中找不到对应的 Token，返回 401。

#### 8.4 登出逻辑

```
logout(token)
  ├─ 解析 Token 获取用户名
  └─ 删除 Redis 中对应的 key
```

登出不需要"让 Token 失效"——JWT 是无状态的，签发后无法撤回。我们能做的是删除 Redis 中的 Token 记录，这样拦截器在下一步校验时会发现 Redis 中没有匹配的 Token，从而拒绝访问。

### 第九步：JWT 工具类 — 双 Token 生成

**操作**：创建 [JwtUtil.java](src/main/java/com/zyt/utils/JwtUtil.java)。

```java
@Component
public class JwtUtil {
    @Value("${jwt.secret}")
    private String secret;
    @Value("${jwt.access-token-expiration}")
    private long accessTokenExpiration;
    @Value("${jwt.refresh-token-expiration}")
    private long refreshTokenExpiration;

    public String generateAccessToken(String username) { ... }
    public String generateRefreshToken(String username) { ... }
    public String getUsernameFromToken(String token) { ... }
    public boolean validateToken(String token) { ... }
}
```

**设计要点**：

- **`@Value` 从配置文件读取密钥和两个过期时间**：accessToken 和 refreshToken 有不同的过期时间，配置独立可灵活调整。
- **`Keys.hmacShaKeyFor()`**：将字符串密钥转为 `SecretKey` 对象。jjwt 0.12.x 要求密钥长度至少 256 位（32 字节）。
- **Token 类型标识**：双 Token 在 JWT claims 中写入 `"type": "access"` 或 `"type": "refresh"`，防止用 refreshToken 当 accessToken 使用。
- **JWT 的 claims（载荷）只存了用户名**：不要存密码、手机号等敏感信息——JWT 的 payload 是 Base64 编码，任何人拿到 Token 都能直接解码查看。签名只防篡改，不防读取。
- **保留 `generateToken()` 兼容方法**：内部调用 `generateAccessToken()`，确保老代码不需要改动。

**双 Token 过期策略**：
- accessToken：30 分钟（`1800000` 毫秒）—— 高频使用，短时效降低泄露风险
- refreshToken：7 天（`604800000` 毫秒）—— 低频使用，长时效避免频繁登录
- Redis 中两个 key 的 TTL 分别与 JWT 过期时间一致，双重保险

### 第十步：统一响应类

**操作**：创建 [ResponseUtil.java](src/main/java/com/zyt/utils/ResponseUtil.java)。

```java
public class ResponseUtil {
    private Integer code;   // 200成功, 400客户端错误, 401未授权
    private String msg;     // 提示信息
    private Object data;    // 携带数据（Token 等）
}
```

**为什么不用 `Map<String, Object>`？** 统一类型便于：
- 前端统一解析（知道一定有三个字段）
- Swagger 文档自动生成准确的返回示例
- 编译期类型安全

**类名叫 `ResponseUtil` 而非 `Result`**：个人习惯，表明这是一个工具性质的统一响应体。

**设计要点**：

- **异步发送（`new Thread`）**：SMTP 连接163服务器需要数秒（SSL握手+认证+投递），同步发送会阻塞 HTTP 响应，前端 Axios 超时（10s）就会报"网络错误"但邮件实际已发出。异步发送让接口在 Redis 存码后立即返回200，邮件后续在后台投递。

### 第十一步：Spring Security 配置 — 只保留密码加密

**操作**：创建 [SecurityConfig.java](src/main/java/com/zyt/config/SecurityConfig.java)。

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {
    @Bean
    public BCryptPasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .csrf(csrf -> csrf.disable())
            .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(auth -> auth.anyRequest().permitAll());
        return http.build();
    }
}
```

**三个关键禁用的原因**：

1. **禁用 CSRF**：CSRF 保护依赖服务端 Session 中的 CSRF Token。我们的 API 是无状态的（stateless），不存 Session，前端每次请求带 JWT 无需 CSRF Token。禁用它避免所有 POST 请求被 403 拦截。

2. **设为无状态会话**：Spring Security 默认会创建 `HttpSession` 存储安全上下文。设为 `STATELESS` 后，每次请求独立认证——这正是 JWT 的使用模式。

3. **放行所有请求**：认证逻辑由 `LoginInterceptor` 处理，Spring Security 的过滤器链不再拦截任何请求。如果不放行，Spring Security 默认会要求所有请求通过认证。

**引入 Spring Security 的唯一目的**：获得 `BCryptPasswordEncoder` 这个 Bean。这样做比直接引入 `spring-security-crypto` 更"标准"，因为大项目中迟早会用到 Spring Security 的其他功能（如方法级权限控制）。

### 第十二步：登录拦截器 — 自定义 Token 校验

**操作**：创建 [LoginInterceptor.java](src/main/java/com/zyt/config/LoginInterceptor.java)。

这是本项目鉴权体系的**核心**，校验链路如下：

```
请求到达
  ├─ 1. 从 Authorization 请求头提取 Token
  │    └─ 为空 → 401 "未携带 Token"
  ├─ 2. JWT 自校验（签名 + 过期）
  │    └─ 无效/过期 → 401 "Token 无效或已过期"
  ├─ 3. 从 JWT 中解析出用户名
  ├─ 4. 用用户名查 Redis，获取当前有效 Token
  │    └─ Redis 中无记录 或 不匹配 → 401 "Token 已失效（被新登录踢掉或已登出）"
  └─ 放行，进入 Controller
```

**为什么用 `HandlerInterceptor` 而非 Spring Security Filter？**
- `HandlerInterceptor` 是 Spring MVC 层面的组件，能拿到已解析的 `HandlerMethod`，可以做更精细的控制。
- 配置更简单——在 `WebConfig` 中注册拦截器即可，不需要理解 Spring Security 的过滤器链顺序。
- 由于已经在 `SecurityConfig` 中关闭了 Spring Security 的所有保护，用拦截器是更直白的写法。

**关于 Token 存储位置**：
- 客户端发送 Token 用 `Authorization` 请求头（而非自定义头如 `X-Token`），这是 HTTP 标准做法，便于后续接入 Swagger、Apifox 等工具时直接使用标准认证配置。
- Redis key 设计为 `token:{username}`：前缀区分命名空间，避免与其他业务 key 冲突。

**拦截器返回 `false` 时**：请求不会进入 Controller，直接返回 401 状态码 + JSON 错误体。

## 第十七步补充A：全局异常处理（@ControllerAdvice）

### 为什么需要全局异常处理？

在引入全局异常处理之前，Service 层的错误返回方式是这样的：

```java
// ❌ 旧写法：Service 中到处是 return new ResponseUtil(400, "xxx", null)
if (Objects.isNull(email) || email.trim().isEmpty()) {
    return new ResponseUtil(400, "邮箱不能为空", null);
}
```

这种写法的**五大痛点**：

1. **Controller 和 Service 职责混乱**：Service 既处理业务逻辑又处理 HTTP 响应格式，违反单一职责原则
2. **错误处理分散**：每个 Service 方法各自决定如何返回错误，没有统一标准——有的返回 400，有的返回 200 但 code=400，维护时需要在十几个方法中逐个查找
3. **Controller 无法区分成功和失败**：`ResponseUtil` 是一个普通对象，Controller 收到它后不知道业务是否成功，只能原样返回
4. **无法利用 Spring 的异常传播机制**：异常可以跨越多层调用栈自动向上传播，而 `return new ResponseUtil(...)` 只能在当前方法返回
5. **不利于 AOP 扩展**：无法用切面统一记录错误日志、发送告警等

**Spring MVC 标准做法**：Service 层遇到错误就**抛异常**，由全局 `@ControllerAdvice` 统一捕获并转换为 JSON 响应。

### 异常体系设计

创建 `com.zyt.exception` 包，建立**两层异常层次结构**：

```
RuntimeException
  └── BusinessException（基类，携带 code + msg + data）
        ├── BadRequestException     → 400 参数校验失败
        ├── UnauthorizedException   → 401 未认证
        ├── ForbiddenException      → 403 权限不足
        ├── NotFoundException       → 404 资源不存在
        └── RateLimitException      → 429 频率限制
```

**操作**：创建 6 个异常类：

**基类 `BusinessException.java`**：

```java
public class BusinessException extends RuntimeException {
    private final Integer code;    // HTTP 业务状态码
    private final Object data;     // 可选的附带数据

    public BusinessException(Integer code, String msg) {
        super(msg);                // msg 存入 RuntimeException，getMessage() 可获取
        this.code = code;
        this.data = null;
    }
    // getter...
}
```

**5 个语义子类**（以 `BadRequestException` 为例）：

```java
public class BadRequestException extends BusinessException {
    public BadRequestException(String msg) {
        super(400, msg);  // code 硬编码，调用方只需传错误描述
    }
}
```

**设计要点**：

- **继承 `RuntimeException` 而非 `Exception`**：运行时异常不需要在方法签名上 `throws`，对现有代码零侵入。Spring 默认只对 `RuntimeException` 做事务回滚。
- **code 在子类中硬编码**：`throw new BadRequestException("邮箱不能为空")` 比 `throw new BusinessException(400, "邮箱不能为空")` 更语义化、更不容易写错状态码。调用方看到异常类名就知道错误的 HTTP 语义。
- **为什么要两层而不是五个独立类？** `GlobalExceptionHandler` 只需写一个 `@ExceptionHandler(BusinessException.class)` 方法就能捕获所有 5 种子类异常，通过 `ex.getCode()` 区分——不用写 5 个 handler 方法。如果后续增加 `ServiceUnavailableException(503)`，handler 代码零改动。
- **`msg` 存入 `RuntimeException.getMessage()`**：如果某处只 catch 了 `Exception` 没处理 `BusinessException`，`e.getMessage()` 仍然能拿到业务错误信息，不会丢失。

### 全局异常处理器

**操作**：创建 `GlobalExceptionHandler.java`。

```java
@RestControllerAdvice  // = @ControllerAdvice + @ResponseBody（返回 JSON 而非视图）
public class GlobalExceptionHandler {

    // ① 业务异常 — 统一入口
    @ExceptionHandler(BusinessException.class)
    public ResponseEntity<ResponseUtil> handleBusinessException(BusinessException ex) {
        ResponseUtil body = new ResponseUtil(ex.getCode(), ex.getMessage(), ex.getData());
        if (ex.getCode() == 401) {
            return ResponseEntity.status(401).body(body);  // 401 必须走真实 HTTP 401
        }
        return ResponseEntity.ok(body);  // 其他异常 HTTP 200，body.code 为实际错误码
    }

    // ② Spring MVC 参数校验异常（预留，配合 @Valid 使用）
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ResponseUtil> handleValidation(MethodArgumentNotValidException ex) { ... }

    // ③ 请求体 JSON 格式错误
    @ExceptionHandler(HttpMessageNotReadableException.class)
    public ResponseEntity<ResponseUtil> handleMessageNotReadable(...) { ... }

    // ④ HTTP 方法错误（GET 调 POST 等）
    @ExceptionHandler(HttpRequestMethodNotSupportedException.class)
    public ResponseEntity<ResponseUtil> handleMethodNotSupported(...) { ... }

    // ⑤ 兜底 — 未预期的运行时异常
    @ExceptionHandler(Exception.class)
    public ResponseEntity<ResponseUtil> handleException(Exception ex) {
        log.error("未捕获的服务器异常", ex);  // 记录完整堆栈
        return ResponseEntity.ok(ResponseUtil.error(500, "服务器内部错误"));
    }
}
```

**设计要点**：

- **`@RestControllerAdvice`**：等于 `@ControllerAdvice` + `@ResponseBody`。被它拦截的异常会直接序列化为 JSON，不经过视图解析器——这正是 REST API 需要的行为。

- **A+B 方案：401 走 HTTP 401，其他走 HTTP 200**：
  
  ```
  BadRequestException("邮箱不能为空")
    → GlobalExceptionHandler
      → ResponseEntity.ok(body)         ← HTTP 200，body.code = 400
        → Axios 成功拦截器: res.code !== 200 → ElMessage.error("邮箱不能为空")  ✅
  
  UnauthorizedException("Token 无效")
    → GlobalExceptionHandler
      → ResponseEntity.status(401).body(body)  ← HTTP 401，body.code = 401
        → Axios 错误拦截器: response.status === 401 → 触发 Token 自动刷新  ✅
  ```
  
  为什么 401 必须走真实 HTTP 401？因为前端 Axios 的**错误拦截器**靠 `response.status === 401` 来触发 Token 自动刷新逻辑。如果 401 也走 HTTP 200，Token 过期后前端不会自动刷新，只会弹一个错误提示。

- **Spring 自动选择最精确的 handler**：如果抛出 `BadRequestException`（继承 `BusinessException`），Spring 会匹配 `@ExceptionHandler(BusinessException.class)` 而非 `@ExceptionHandler(Exception.class)`——因为前者更精确。这保证了业务异常走 handler ①，真正的未知异常走 handler ⑤。

- **兜底 handler 不暴露异常细节**：`@ExceptionHandler(Exception.class)` 返回固定文案 "服务器内部错误"，绝不把 `NullPointerException: at com.zyt.service.UserServiceImpl.login(UserServiceImpl.java:247)` 这样的信息返回给客户端——这是安全基本要求。完整堆栈通过 `log.error()` 记录到服务器日志。

### 拦截器的错误响应 — `toJsonError()`

**关键认知**：拦截器在请求进入 Controller **之前**执行，因此拦截器中抛出的异常**不会被** `@ControllerAdvice` 捕获。拦截器必须自己 `write()` 错误 JSON 到 `HttpServletResponse`。

旧代码用字符串拼接构造 JSON：

```java
// ❌ 旧写法：字符串拼接，msg 中包含双引号就会破坏 JSON 结构
writer.write("{\"code\":401,\"msg\":\"" + msg + "\",\"data\":null}");
```

msg 是 "Token 已失效，请重新登录" 没问题，但如果 msg 中包含 `"`（如用户在密码里输了引号），JSON 就坏了。

**操作**：在 `ResponseUtil` 中添加 `toJsonError()` 静态方法：

```java
private static final ObjectMapper MAPPER = new ObjectMapper();

public static String toJsonError(int code, String msg) {
    try {
        Map<String, Object> map = new LinkedHashMap<>();
        map.put("code", code);
        map.put("msg", msg);      // Jackson 自动转义特殊字符
        map.put("data", null);
        return MAPPER.writeValueAsString(map);
    } catch (Exception e) {
        return "{\"code\":500,\"msg\":\"服务器内部错误\",\"data\":null}";
    }
}
```

然后 `LoginInterceptor.writeError()` 改为：

```java
writer.write(ResponseUtil.toJsonError(401, "Token 无效或已过期"));
```

`RoleInterceptor.writeError()` 改为：

```java
writer.write(ResponseUtil.toJsonError(403, "权限不足，需要 admin 权限"));
```

**为什么用 Jackson 而不是 fastjson/Gson？** Spring Boot 默认使用 Jackson 做 JSON 序列化，`ObjectMapper` 已经在 Spring 容器中。直接复用避免了引入第二个 JSON 库。`LinkedHashMap` 保证字段输出顺序为 code → msg → data，与 `ResponseUtil` 的字段顺序一致。

**为什么要保留 `new ResponseUtil(code, msg, data)` 构造器？** 兼容性——Controller 中还有大量 `return new ResponseUtil(200, "成功", data)` 的写法，这些不需要改动。`toJsonError()` 只给拦截器用。

### Service 层改造：从 return 到 throw

**操作**：将 `UserServiceImpl` 中部分参数校验改为抛异常。

```java
// ✅ 新写法：抛异常，由 @ControllerAdvice 统一处理
if (Objects.isNull(email) || email.trim().isEmpty()) {
    throw new BadRequestException("邮箱不能为空");
}

if (StringUtils.isEmpty(password) || password.length() < 6) {
    throw new BadRequestException("密码长度不能小于6位");
}
```

**为什么没有改造所有方法？** 改造成本是逐步的——`sendCode()` 和 `register()` 的 13 处参数校验已改为 `throw`，涉及 Redis 操作的复杂控制流（如登录的暴力破解判断、微信回调的状态流转）仍保留 `return new ResponseUtil(...)` 模式。两种写法在架构上不冲突：抛异常由 `@ControllerAdvice` 接管，`return ResponseUtil` 由 Controller 原样返回。后续重构时可以逐步统一。

## 第十七步补充B：Knife4j / SpringDoc 接口文档

### 为什么需要接口文档？

项目有 18 个 REST 端点分布在 4 个功能模块中。没有文档时：

- 前端开发者需要翻 Controller 源码才能知道参数名和响应格式
- 测试人员用 curl/Postman 手动拼 URL 和参数
- 接口变更后没有机制通知所有消费者
- 新人接手项目时完全没有"系统有哪些 API"的全局视图

**OpenAPI 3 标准**：业界通用的 API 描述规范（原 Swagger 2），用 JSON/YAML 描述接口的路径、参数、响应结构。SpringDoc 是 OpenAPI 3 在 Spring Boot 上的实现——它自动扫描 `@RestController` 并生成 OpenAPI JSON。

**Knife4j** 是国内最流行的 SpringDoc UI 增强方案：它在 SpringDoc 生成的 OpenAPI JSON 之上，提供了一个更好用的 Web UI（`/doc.html`），支持：
- 按 `@Tag` 分组展示接口（比原生 Swagger UI 的单一列表清晰得多）
- 全局 Authorization 配置（填一次 Token 所有接口自动携带）
- 离线文档导出（Markdown / HTML / OpenAPI JSON）
- 在线调试（直接在浏览器里调接口，自动处理参数）

### 技术选型：Knife4j 4.x + SpringDoc OpenAPI 3

| 方案 | 问题 | 结论 |
|------|------|------|
| 原生 SpringDoc + Swagger UI | UI 体验差，没有分组折叠、没有全局参数、没有离线导出 | ❌ |
| Knife4j 3.x（OpenAPI 2） | Spring Boot 3.x 使用 Jakarta 命名空间，需要 OpenAPI 3 | ❌ |
| **Knife4j 4.x（OpenAPI 3 + Jakarta）** | 完美适配 Spring Boot 3.x，功能全面 | ✅ |

### 第一步：添加依赖

**操作**：在 `pom.xml` 中添加：

```xml
<dependency>
    <groupId>com.github.xiaoymin</groupId>
    <artifactId>knife4j-openapi3-jakarta-spring-boot-starter</artifactId>
    <version>4.5.0</version>
</dependency>
```

**为什么是 `jakarta` 版本？** Spring Boot 3.x 将 `javax.*` 命名空间迁移到了 `jakarta.*`（Jakarta EE 9+）。普通的 `knife4j-openapi3-spring-boot-starter` 依赖的是 `javax.servlet`，在 Spring Boot 3.x 下会报 `NoClassDefFoundError`。

### 第二步：配置文件

**操作**：在 `application.yml` 末尾添加：

```yaml
springdoc:
  swagger-ui:
    path: /swagger-ui.html
  api-docs:
    path: /v3/api-docs           # OpenAPI JSON 端点
  group-configs:
    - group: default
      paths-to-match: /user/**   # 只扫描 /user 下的接口
      packages-to-scan: com.zyt.controller

knife4j:
  enable: true
  setting:
    language: zh-CN              # Knife4j UI 中文界面
    enable-swagger-models: true  # 显示 Schema 模型
    enable-document-manage: true # 启用离线文档导出
```

**`paths-to-match: /user/**` 的作用**：限制文档只展示 `/user` 下的接口。如果不配置，Spring Boot 内嵌的 `/actuator/health`、`/error` 等也会出现在文档中，造成干扰。

### 第三步：OpenAPI 配置 Bean

**操作**：创建 `Knife4jConfig.java`。

```java
@Configuration
public class Knife4jConfig {
    @Bean
    public OpenAPI customOpenAPI() {
        return new OpenAPI()
                .info(new Info()
                    .title("Spring Boot Auth 用户认证系统")
                    .version("1.0")
                    .description("...Markdown 格式的完整系统说明..."))
                // 全局 Authorization — 核心功能
                .addSecurityItem(new SecurityRequirement().addList("Authorization"))
                .components(new Components()
                    .addSecuritySchemes("Authorization", new SecurityScheme()
                        .name("Authorization")
                        .type(SecurityScheme.Type.APIKEY)
                        .in(SecurityScheme.In.HEADER)
                        .description("JWT accessToken（直接填入，无需 Bearer 前缀）")));
    }
}
```

**设计要点——为什么用 `SecurityScheme.Type.APIKEY` 而非 `HTTP`？**

这是 Knife4j 配置中最容易踩的坑：

| Scheme 类型 | Knife4j 注入行为 | 本项目是否适用 |
|-------------|-----------------|:---:|
| `HTTP` (Bearer) | 自动加 "Bearer " 前缀：`Authorization: Bearer xxx` | ❌ 后端解析失败 |
| **`APIKEY`** | 原样填入：`Authorization: xxx` | ✅ 与拦截器解析方式一致 |

本项目的 `LoginInterceptor` 直接从 `Authorization` header 取字符串作为 JWT Token，不需要也不识别 "Bearer " 前缀。所以必须用 `APIKEY`。

**`addSecurityItem` + `addSecuritySchemes` 是如何配合的？**
- `addSecuritySchemes` 定义一个名为 "Authorization" 的认证方式（header 名是什么、类型、提示文字）
- `addSecurityItem` 将这个认证方式应用到所有接口
- Knife4j UI 渲染时，右上角会出现一个 **Authorize** 按钮 → 点击弹出输入框 → 填入 Token → 所有接口的"发送"按钮自动带上这个 header

### 第四步：WebConfig 放行 Knife4j 路径

**操作**：在 `WebConfig` 中追加排除路径。

Knife4j 需要以下 6 个路径不受拦截器限制：

| 路径 | 用途 |
|------|------|
| `/doc.html` | Knife4j UI 主页 |
| `/webjars/**` | Knife4j 前端 JS/CSS 静态资源（从 Maven 依赖中加载） |
| `/v3/api-docs/**` | OpenAPI 3 JSON 文档端点 |
| `/swagger-ui/**` | 原生 Swagger UI（保留兼容） |
| `/swagger-resources/**` | Swagger 资源路径 |
| `/favicon.ico` | 浏览器标签页图标 |

```java
String[] swaggerPaths = {"/doc.html", "/webjars/**", "/v3/api-docs/**",
        "/swagger-ui/**", "/swagger-resources/**", "/favicon.ico"};

// 两个拦截器各自追加排除
registry.addInterceptor(loginInterceptor)
        .excludePathPatterns(swaggerPaths);  // ← 原有基础上追加
registry.addInterceptor(roleInterceptor)
        .excludePathPatterns(swaggerPaths);  // ← 原有基础上追加
```

如果不放行，访问 `/doc.html` 时会被 `LoginInterceptor` 拦截要求登录——文档本身就要告诉用户怎么登录，所以文档页面必须公开。

### 第五步：Controller 添加 Swagger 注解

**操作**：为 `UserController` 的 18 个端点添加注解。

#### 分组策略：类级默认 + 方法级覆盖

```java
@RestController
@RequestMapping("/user")
@Tag(name = "基础认证")  // ← 类级默认：7 个不单独标记 Tag 的方法归入此组
public class UserController {

    // ① 基础认证 — 使用类级默认 Tag，不需要方法级 @Tag
    @Operation(summary = "获取图形验证码", description = "...")
    @GetMapping("/captcha")
    public ResponseUtil captcha() { ... }

    // ② 微信扫码登录 — 方法级 @Tag 覆盖类级默认
    @Tag(name = "微信扫码登录")
    @Operation(summary = "生成微信扫码场景", description = "...")
    @PostMapping("/wechat/qrcode")
    public ResponseUtil wechatQrcode() { ... }

    // ③ 邮箱找回密码 — 方法级覆盖
    @Tag(name = "邮箱找回密码")
    @Operation(summary = "发送密码重置验证码", description = "...")
    @PostMapping("/forgot-password/send-code")
    public ResponseUtil sendResetCode() { ... }

    // ④ RBAC — 方法级覆盖
    @Tag(name = "RBAC 角色权限")
    @Operation(summary = "分页查询所有用户（管理员）", description = "...")
    @RequireRole("admin")
    @GetMapping("/admin/users")
    public ResponseUtil listUsers() { ... }
}
```

**为什么用方法级 `@Tag` 覆盖而不是拆 4 个 Controller？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| 拆 4 个 Controller | 每个 Controller 职责单一 | 注册路径分散，`/user` 前缀下的端点分布在 4 个文件中，查找不便 |
| **类级默认 + 方法级覆盖** | 所有 `/user/**` 端点在一处，Knife4j 自动分组 | 一个 Controller 文件较长（~160行） |

18 个端点共 160 行，一个文件完全可控。如果以后增长到 50+ 端点再考虑拆 Controller。

#### 特殊端点处理

| 端点 | 处理方式 | 原因 |
|------|---------|------|
| `/wechat/callback` | `@Operation(hidden = true)` | 返回 HTML 页面（给手机浏览器看），不是 JSON API，出现在文档中会造成困惑 |
| `/logout` | `@Parameter` 标记 Header 参数 | Token 从 `Authorization` header 读取，不是 RequestBody，必须明确标注 |
| `/admin/users` | 保留 `@RequireRole("admin")` | 文档不拦截请求，但注解本身标记了权限要求，开发者一看就明白 |

#### 参数注解模式

```java
// 路径参数（Query String / Path Variable）
@Parameter(description = "场景 ID", required = true, example = "uuid-string")
@RequestParam String sceneId

// Header 参数
@Parameter(description = "JWT accessToken", required = true)
@RequestHeader("Authorization") String token

// 分页参数
@Parameter(description = "页码", example = "1") @RequestParam(defaultValue = "1") int page
@Parameter(description = "每页条数", example = "10") @RequestParam(defaultValue = "10") int size
```

`example` 参数在 Knife4j UI 中直接作为默认值预填，在线调试时不需要手动输入。

### 第六步：实体类注解

**操作**：`User.java` 的 `password` 字段加 `@Schema(hidden = true)`。

```java
@Schema(hidden = true)
private String password;
```

**为什么隐藏 password？** 即使数据库中存的是 BCrypt 密文（不可逆），在文档的 Schema 展示中暴露密码字段仍然是不安全的做法——前端开发者可能误以为可以传入 raw password，或者日志中意外打印了密文。隐藏后，Swagger 的 Schemas 面板不会显示 password 字段。

### Knife4j vs 手写文档

| 维度 | 手写 Markdown/Word 文档 | Knife4j 自动生成 |
|------|------------------------|-------------------|
| 准确性 | 接口改了文档经常忘了更新 | 扫描实际代码生成，100% 准确 |
| 调试能力 | 只能看，不能调 | 浏览器里直接发请求，即时看到响应 |

### 已知兼容性问题：Spring Boot 3.5.x + Knife4j 4.5.0

**问题**：访问 `http://localhost:8080/doc.html` 时左侧菜单缺少「接口文档」项，`/v3/api-docs/default` 返回 500。

**原因**：Spring Framework 6.2（Spring Boot 3.5.x）中 `ControllerAdviceBean` 构造函数从单参数改为三参数，Knife4j 4.5.0 内嵌的 springdoc-openapi 版本太旧，仍在调用已删除的旧构造函数：

```
NoSuchMethodError: ControllerAdviceBean.<init>(Object)
```

**当前修复**（2026-06-03）：

# 第二部分：前端（Vue 3 + Element Plus）

## 前言：为什么做前后端分离

在本教程第一部分中，后端已完成了完整的 RESTful API（注册、登录、登出、鉴权）。但我们没有在后端工程里直接写前端页面——Spring Boot 的 `static` 目录虽然可以托管静态文件，但在实际开发中存在几个问题：

1. **前后端耦合**：改前端样式也要重新编译打包整个后端项目
2. **开发体验差**：改一行 CSS 要等 Maven 重新构建，没有热更新
3. **团队协作困难**：前端和后端开发者在同一仓库里互相等待
4. **部署不灵活**：前端静态文件适合放 CDN 或 Nginx，不适合放在 Java 进程里

所以我们将前端独立为一个 **Vue 3 + Vite + Element Plus** 工程，与后端 **完全分离**。

## 第十八步：初始化 Vue 3 + Vite 项目

**操作**：在项目根目录执行：

**为什么用 Vite 而不是 Vue CLI？**
- Vue CLI 基于 Webpack，项目大了之后开发服务器启动慢、热更新慢
- Vite 基于原生 ES Module，开发服务器秒启动，HMR 几乎瞬间生效
- Vite 8.x 使用 Rolldown（Rust 编写的打包器）做生产构建，速度远超 Webpack

**为什么 `frontend` 放在后端项目根目录下？**
- 同一个 Git 仓库便于管理，一套 `git clone` 拿到前后端全部代码
- 但它们是两个独立工程，各有各的 `package.json` 和 `pom.xml`，互不依赖

## 第二十步：配置 Vite — 代理与按需导入

**三个关键配置的缘由**：

3. **`ElementPlusResolver`**：告诉 `unplugin-vue-components` 去 Element Plus 包里找组件。它的工作流程是：
   - 扫描 `.vue` 文件中的模板标签
   - 发现 `<el-button>` → 自动添加 `import { ElButton } from 'element-plus'` 
   - 同时自动引入对应的 CSS 样式文件

   > 如果不用这个 resolver，每次用新组件都要手动 `import` 并在 `components` 中注册。

## 第二十一步：Token 管理工具 — 双 Token 存储

**操作**：创建 [token.js](frontend/src/utils/token.js)。

```javascript
const ACCESS_TOKEN_KEY = 'zyt_access_token'
const REFRESH_TOKEN_KEY = 'zyt_refresh_token'

export function getToken() { return localStorage.getItem(ACCESS_TOKEN_KEY) }
export function getRefreshToken() { return localStorage.getItem(REFRESH_TOKEN_KEY) }

export function setTokens(accessToken, refreshToken) {
  localStorage.setItem(ACCESS_TOKEN_KEY, accessToken)
  localStorage.setItem(REFRESH_TOKEN_KEY, refreshToken)
}
export function setToken(token) { localStorage.setItem(ACCESS_TOKEN_KEY, token) }
export function removeToken() {
  localStorage.removeItem(ACCESS_TOKEN_KEY)
  localStorage.removeItem(REFRESH_TOKEN_KEY)
}
```

**为什么用两个 key 分别存储？**
- accessToken 是高频读写的（每次请求都要从 localStorage 读取），单独存避免解析 JSON 的开销
- refreshToken 只在自动刷新时读取一次，与 accessToken 生命周期解耦
- `removeToken()` 同时清除两个 Token，保证登出时完全清理

**为什么用 localStorage 存 Token？**

| 方案 | 优点 | 缺点 |
|------|------|------|
| Cookie | 自动随请求发送 | 有 CSRF 风险，需要额外配置 SameSite |
| sessionStorage | 关闭标签页自动清除 | 用户刷新页面就丢失，不实用 |
| **localStorage** | 持久化存储，跨标签页共享 | XSS 可读取（但 Vue 默认转义输出，风险可控） |

我们选择 `localStorage`：前端 SPA 中 Vue 会自动转义所有模板变量，XSS 攻击面很小。存储键名加了前缀 `zyt_` 以避免与其他项目冲突。

## 第二十二步：Axios 请求层 — 拦截器机制

**操作**：创建 [api/index.js](frontend/src/api/index.js)，这是前端与后端通信的**核心枢纽**。

```javascript
import axios from 'axios'
import { getToken, removeToken } from '@/utils/token'
import { ElMessage } from 'element-plus'

const request = axios.create({
  baseURL: '/',
  timeout: 10000,

```

### 请求拦截器 — 自动带 Token

```javascript
request.interceptors.request.use(
  (config) => {
    const token = getToken()
    if (token) {
      config.headers.Authorization = token
    }
    return config
  },
  (error) => Promise.reject(error),
)
```

**为什么要在拦截器里统一加 Token 而不是在每个请求里手动加？**
- 集中管理：将来如果改用 `Bearer xxx` 格式，只需改一处
- 不会漏加：开发者调用 API 时不需要记得传 Token
- 自动判断：有 Token 就加，没有就不加（注册和登录时还没有 Token）

### 响应拦截器 — 统一错误处理 + 自动无感刷新

```javascript
let isRefreshing = false
let pendingRequests = []

request.interceptors.response.use(
  (response) => {
    const res = response.data
    if (res.code !== 200) {
      ElMessage.error(res.msg || '请求失败')
      return Promise.reject(new Error(res.msg))
    }
    return res
  },
  async (error) => {
    const { config, response } = error
    // 401 时尝试自动刷新 Token
    if (response && response.status === 401 && !config._retry) {
      const refreshToken = getRefreshToken()
      if (!refreshToken) { /* 无 refreshToken，直接踢到登录页 */ }
      // 如果正在刷新中，将请求排队等待
      if (isRefreshing) {
        return new Promise((resolve) => {
          addPendingRequest((newToken) => {
            config.headers.Authorization = newToken
            resolve(request(config))  // 用新 Token 重试
          })
        })
      }
      isRefreshing = true
      try {
        const res = await axios.post('/user/refresh', { refreshToken })
        setToken(res.data.data.accessToken)
        onRefreshed(res.data.data.accessToken)  // 重放所有排队的请求
        config.headers.Authorization = newAccessToken
        return request(config)  // 重试当前请求
      } catch {
        removeToken()
        window.location.href = '/login'
      } finally { isRefreshing = false }
    }
    ...
  },
)
```

**自动刷新机制的完整流程**：

```
请求 → 后端返回 401
  ├─ 无 refreshToken → 直接踢到登录页
  ├─ 有 refreshToken，且当前没有在刷新中
  │    ├─ 标记 isRefreshing = true
  │    ├─ 调用 /user/refresh 获取新 accessToken
  │    ├─ 更新 localStorage 中的 accessToken
  │    ├─ 重放所有排队的请求（用新 Token）
  │    └─ 重试当前请求
  └─ 有 refreshToken，但已经在刷新中
       └─ 将当前请求加入 pendingRequests 队列，等刷新完成后自动重试
```

**为什么需要请求队列（`pendingRequests`）？** 假设页面同时发了 3 个请求，它们都因 accessToken 过期而返回 401。如果不做排队，3 个请求会同时尝试刷新 Token——导致多次刷新、浪费资源、可能出现竞态条件。用 `isRefreshing` 标志 + 队列，确保同一时间只有一个刷新请求，其他请求等待共享结果。

**`config._retry` 的作用**：防止重试后仍然 401 导致死循环。已重试过的请求不再触发刷新逻辑。

**两层错误处理的设计逻辑**：

1. **HTTP 200 但业务码非 200**（如"用户名已存在"、"密码错误"）：后端返回 `{ code: 400, msg: "密码错误" }`，拦截器直接弹出错误提示。

2. **HTTP 401**（accessToken 过期）：自动用 refreshToken 无感刷新，用户完全无感知。只有 refreshToken 也过期时才跳转登录页。

3. **网络错误**（后端宕机、超时等）：提示通用错误信息。

### 懒加载

`() => import('@/views/Login.vue')` 而非 `import Login from '@/views/Login.vue'`：
- 静态 `import` 会把所有页面打包到一个 JS 文件，首屏加载慢
- 动态 `import()` 让 Vite 把每个页面拆成独立的 chunk，只有访问时才加载
- 三个页面 → 三个独立 JS 文件，首屏只下载当前页面需要的那一个

### 导航守卫 — 前端鉴权

```javascript
router.beforeEach((to, from, next) => {
  const token = getToken()
  if (to.meta.requiresAuth && !token) {
    next('/login')        // 未登录 → 去登录
  } else if (to.meta.guest && token) {
    next('/home')          // 已登录 → 不让你看登录页
  } else {
    next()                 // 放行
  }
})
```

**三层判断**：

| 场景 | Token | 目标页 | 结果 |
|------|-------|--------|------|
| 未登录访问首页 | 无 | `/home` (requiresAuth) | → 跳到 `/login` |
| 已登录访问登录页 | 有 | `/login` (guest) | → 跳到 `/home` |
| 正常访问 | 任意 | 匹配 | 放行 |

> **注意**：前端路由守卫只是**用户体验优化**（避免用户看到空白页），真正的安全防护在后端 `LoginInterceptor`。攻击者可以直接用 curl 绕过前端，但绕不过后端的拦截器。

### 登录流程

```javascript
const handleLogin = async () => {
  const valid = await formRef.value.validate().catch(() => false)
  if (!valid) return
  loading.value = true
  try {
    const res = await login(form.username, form.password, captchaId.value, form.captchaCode)
    setTokens(res.data.accessToken, res.data.refreshToken)
    ElMessage.success('登录成功')
    router.push('/home')
  } catch {
    refreshCaptcha()  // 登录失败自动刷新验证码
  } finally {
    loading.value = false
  }
}
```

**`onMounted` 时自动获取验证码**，页面加载即有验证码图片。登录失败后 `catch` 中调用 `refreshCaptcha()` 刷新——因为验证码是一次性使用的，失败时必须换新的。

**关键细节**：
- `loading.value = true` → 按钮显示「登录中...」并禁用，防止重复提交
- `finally` 中关闭 loading——即使出错也要恢复按钮，否则按钮永远灰掉
- 支持回车键提交：`@keyup.enter="handleLogin"` 绑定在 `el-form` 上

### 验证码发送 + 60秒倒计时

```javascript
const handleSendCode = async () => {
  if (!form.email || !emailReg.test(form.email)) {
    ElMessage.warning('请先输入正确的邮箱地址')
    return
  }
  if (!form.captchaCode) {
    ElMessage.warning('请先输入图形验证码')
    return
  }
  await sendCode(form.email, captchaId.value, form.captchaCode)
  ElMessage.success('验证码已发送，请查看邮箱')
  countdown.value = 60
  countdownTimer = setInterval(() => {
    countdown.value--
    if (countdown.value <= 0) clearInterval(countdownTimer)

}
```

```javascript
const handleLogout = async () => {
  await ElMessageBox.confirm('确定要退出登录吗？', '提示', {
    confirmButtonText: '确定', cancelButtonText: '取消', type: 'warning',
```

## 前端数据流示意（含自动刷新）

```
用户操作 → Vue 组件 → api/user.js → axios instance
                                          │
                         请求拦截器：从 localStorage 读取 accessToken
                         加到 Authorization 请求头
                                          │
                                    HTTP 请求 → 后端
                                          │
                              ┌── 200 正常响应 → 返回 data
                              │
                              ├── 401（accessToken 过期）
                              │     ├─ 取 refreshToken → 调 /user/refresh
                              │     ├─ 拿到新 accessToken → 更新 localStorage
                              │     ├─ 重放排队的请求
                              │     └─ 重试当前请求（用户无感知）
                              │
                              └── refreshToken 也过期
                                    └─ 清 Token → 跳转登录页
```

## 开发中的常见问题与解决方案

### Q1：`npm run dev` 后页面空白

**原因**：Vite 构建缓存问题或 Vite 版本不兼容。

**解决**：删除 `node_modules` 和 `package-lock.json`，重新 `npm install`。

### Q2：登录后立刻被跳回登录页

**原因**：路由守卫检测不到 Token。检查：
1. 浏览器 DevTools → Application → Local Storage 中是否有 `zyt_access_token` 和 `zyt_refresh_token`
2. 后端登录接口是否真的返回了双 Token（检查 Network 标签的响应体）

### Q3：请求发到了 3000 端口而不是 8080

**原因**：代理只对开发服务器生效。检查：
1. `axios` 的 `baseURL` 设置为 `/` 而不是 `http://localhost:8080`
2. Vite dev server 必须通过 `npm run dev` 启动，直接打开 HTML 文件不行

### Q4：Token 自动刷新机制是如何工作的？

1. accessToken 过期（30 分钟）→ 后端返回 401
2. Axios 响应拦截器捕获 401，用 refreshToken 自动请求 `/user/refresh`
3. 获得新 accessToken，更新 localStorage，用新 Token 重试原请求
4. 如果同时有多个请求返回 401，第一个会触发刷新，其余排队等待
5. 只有当 refreshToken 也过期（7 天）时，用户才需要重新登录

### Q5：页面样式异常，按钮没有样式

**原因**：Element Plus CSS 未引入。

**解决**：确认 `main.js` 中有 `import 'element-plus/dist/index.css'`。

## 项目架构分层（全栈）

```
浏览器 → Vue Router → Axios ──HTTP──→ LoginInterceptor → Controller → Service → Mapper → Database
                │                          │                    ↕          ↕
           localStorage              JWT + Redis             JwtUtil     Redis
        (accessToken +                   │
         refreshToken)            token:{username}
                                  refresh_token:{username}
```

**后端每层的职责边界**：
- **Controller**：解析 HTTP 参数，调用 Service，返回 JSON。不应包含任何业务逻辑。
- **Service**：业务逻辑 + 事务管理。本项目未加 `@Transactional`，因为都是单表操作。
- **Mapper**：数据访问，MyBatis-Plus 自动生成，不需要手写。
- **Config**：拦截器 / 安全配置 / 分页插件，相当于项目的"基础设施"。
- **Utils**：纯工具类，JWT 和统一响应，无业务依赖。

**前端每层的职责边界**：
- **Views**：页面组件，专注 UI 交互和表单校验，调用 API 层获取数据
- **API**：Axios 实例 + 请求/响应拦截器（含自动刷新） + 接口方法封装
- **Router**：路由配置 + `beforeEach` 导航守卫，实现前端鉴权
- **Utils**：双 Token 存取工具，桥接 API 层和 localStorage

## 关键设计决策速查表

| 决策 | 选择 | 理由 |
|------|------|------|
| 前后端架构 | 完全分离 | 前端独立工程，Vite 代理解决跨域，后端零前端代码 |
| Token 策略 | 双 Token（access + refresh） | 短期 accessToken 降低泄露风险，长期 refreshToken 无需频繁登录 |
| Token 类型校验 | JWT `type` claim | `/refresh` 接口强制校验 `type=refresh`，防 accessToken 无限续期（P0） |
| 无感刷新 | Axios 拦截器 + 请求队列 | 401 时自动用 refreshToken 刷新，多请求并发时排队防止重复刷新 |
| 暴力破解防护 | Redis INCR + 5次锁定15分钟 | 原子计数，TTL 仅首次设置防绕过，统一错误提示防用户枚举（P0） |
| 认证方式 | 自定义拦截器 + JWT | 比 Spring Security 更简单直白，适合 RESTful API |
| 密码存储 | BCrypt | 自带随机盐，每密文不同，抗彩虹表 |
| 单设备登录 | Redis 存双 Token | 利用 Redis TTL 天然过期，读写快，支持同时失效两个 Token |
| 登出实现 | 删 Redis key | JWT 无法主动失效，只能通过服务端额外存储来实现撤销 |
| CSRF | 显式禁用 | JWT 认证下 CSRF 不存在（不依赖 Cookie 传认证信息） |
| 会话管理 | 无状态 | RESTful API 的标准做法，服务器不存用户 Session |
| ORM | MyBatis-Plus | MyBatis 的灵活性 + JPA 式单表 CRUD 免编写 |
| 数据库字符集 | utf8mb4 | 支持 emoji，避免 utf8(3字节) 的坑 |
| 前端 Token 存储 | localStorage | 简单直接，SPA 应用中无 XSS 风险（Vue 自动转义） |
| UI 组件 | Element Plus 按需导入 | `unplugin-vue-components` 自动注册，减小打包体积 |
| 登录页布局 | 双栏左右布局 + 选项卡切换 | 左侧表单区 + 右侧授权/权益信息区，移动端自动上下布局 |
| 微信登录 | 真实 OAuth 2.0 + 邮箱绑定 | 微信测试号免费使用，ngrok 内网穿透，扫码→绑定→登录完整闭环 |
| 图形验证码 | Java AWT 绘制 + Redis 2min TTL | JDK 内置，干扰线+噪点+字符旋转，一次性使用 |
| 邮箱验证码 | 163邮箱 SMTP + Redis 5min TTL | 个人开发者免费可用，Redis 自然过期无需手动清理 |
| 防刷策略 | 图形验证码 + Redis TTL 检查（60秒间隔）+ 暴力破解锁定 | 三重防护：图形验证码阻止机器，60秒限制阻止脚本，5次锁定阻止字典攻击 |
| 颜色主题 | 绿色 `#31c27c` | 清新安全感的品牌色，与 BCrypt/Token 安全机制传达一致 |
| 开发代理 | Vite proxy | 开发时 `/user` 请求代理到 8080，避免跨域问题 |

|---------------|------|----------------|
| `jdbc:mysql://localhost:3306/...` | `jdbc:mysql://${MYSQL_HOST:localhost}:${MYSQL_PORT:3306}/...` | `MYSQL_HOST=mysql` |
| `username: root` | `username: ${MYSQL_USERNAME:root}` | `MYSQL_USERNAME=springuser` |
| `password: 123456` | `password: ${MYSQL_PASSWORD:123456}` | `MYSQL_PASSWORD=springpass` |
| `host: localhost` (Redis) | `host: ${REDIS_HOST:localhost}` | `REDIS_HOST=redis` |
| `jwt.secret` | `secret: ${JWT_SECRET:...}` | `JWT_SECRET=change-me` |
| `spring.mail.username` | `username: ${MAIL_USERNAME:}` | `MAIL_USERNAME=...` |

`${VAR:default}` 语法确保本地 `mvn spring-boot:run` 仍能正常运行（使用默认值），Docker 环境通过 `.env` 覆盖。

### 使用方式

```bash
# 一键构建 + 启动
docker compose up -d --build

# 查看状态
docker compose ps

# 访问
# 前端:      http://localhost
# API 文档:   http://localhost/doc.html
# 健康检查:   http://localhost:8080/actuator/health

# 可选：Adminer 数据库管理
docker compose --profile debug up -d     # → http://localhost:8081

# 停止
docker compose down                       # 保留数据卷
docker compose down -v                    # 删除数据库/Redis 数据
```

### 架构设计要点

- **多阶段构建**：编译阶段用完整 JDK+Maven/Node，运行阶段只用最小 JRE/Nginx 镜像，最终镜像缩小 ~300MB
- **健康检查**：添加 `spring-boot-starter-actuator` 提供 `/actuator/health`，Docker 通过它判断服务是否就绪
- **MySQL 初始化**：挂载 `init.sql` 到 `/docker-entrypoint-initdb.d/`，容器首次启动自动建库建表
- **非 root 用户**：后端容器以 `appuser` 运行，前端以 nginx 默认用户运行
- **AI 扩展预留**：基础 compose 文件保持简洁，未来 AI 服务通过 `docker-compose.ai.yml` 叠加；Nginx 支持 `include conf.d/*.conf` 路由扩展
